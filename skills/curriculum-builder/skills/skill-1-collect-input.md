# Skill 1: 입력 수집·정리

이후 모든 스킬이 참조하는 단일 정보원(SSOT)인 `input.json`을 생성한다.

## 초기 안내 (스킬 시작 시 최우선)

Skill 1이 시작되면 **가장 먼저** 아래 초기 안내를 사용자(LD)에게 표시한다.
이 안내를 생략하고 바로 입력 추출로 넘어가지 않는다.
(같은 실행 흐름 안에서는 재표시하지 않는다. 새 세션에서 /curriculum-builder를 다시 호출할 때만 다시 표시.)

```
👋 커리큘럼 자동 생성을 시작합니다

대화창에 직접 입력하셔도 좋고, 고객사 미팅로그나 별도 문서에 아래 내용이 있으면 그 문서를 그대로 첨부해도 됩니다.

[필수 — 없으면 진행할 수 없습니다]
① 회사명 (예: LG전자)
② 직무 (예: 온라인기획팀, 마케터)
③ 교육에 사용할 툴 (예: ChatGPT, Claude Code — 복수 가능)
④ 교육 주제 (예: "데이터 분석 및 대시보드 구축")

[선택 — 입력해두면 좋지만, 지금 만드는 표준 커리큘럼에는 반영되지 않습니다. 이후 기업 맥락·시수·보안 제약·수강생 수준을 반영해 고도화하는 자유 대화 단계에서 활용됩니다. 미입력 시 기본값 적용]
⑤ 시수 (예: "8시간", "하루 4시간씩 2일" / 미입력 시 기본값: 8시간)
⑥ 보안·환경 제약 (예: "외부 AI 사이트 접근 불가", "사내 데이터 반출 금지" / 미입력 시 기본값: "없음")
⑦ 수강생 수준 (예: "ChatGPT 기본 이해 있음" / 미입력 시 기본값: "평범한 직장인으로서 ChatGPT/Gemini 무료 버전 정도 사용 경험")
```

## 입력 소스

사용자가 대화창에 직접 입력하거나, 고객사 미팅로그·문서·메일 등을 전달할 수 있다.
어떤 형태로 오든 아래 필수/선택 항목을 추출한다.

## 필수 항목 (하나라도 없으면 중단)

| 항목 | 키 | 타입 | 설명 | 예시 |
|------|-----|------|------|------|
| 회사명 | `company` | string | 교육 대상 기업 | "LG전자" |
| 직무 | `role` | string | 교육 대상 직무/팀 | "온라인기획팀" |
| 툴 | `tools` | string[] | 교육에 사용할 생성형 AI 툴(복수 가능) | `["ChatGPT", "Claude Code"]` |
| 주제 | `topic` | string | 교육 주제/방향 | "데이터 분석 및 대시보드 구축" |

## 선택 항목 (미입력 시 기본값 적용)

| 항목 | 키 | 기본값 | 비고 |
|------|-----|--------|------|
| 시수 | `hours` | `8` | 대부분의 요청에는 시수가 포함됨. 정말 입력이 없을 때만 기본값 8시간 적용 |
| 보안 제약 | `security` | `"없음"` | 예: "외부 AI 사이트 접근 불가", "사내 데이터 반출 금지" 등 |
| 수강생 수준 | `level` | `"평범한 직장인으로서 ChatGPT/Gemini 무료 버전 정도 사용 경험"` | 사전 지식 수준 |

## 처리 규칙 (중요)

### 규칙 1: 회사/직무 분리 (보수적)

사용자가 "LG전자 온라인기획팀"처럼 회사와 조직단위를 붙여서 입력하는 경우가 많다. 아래 규칙에 따라 분리한다.

**분리한다**: 회사명 뒤에 조직 하위단위 단어가 붙은 경우.
- 조직 하위단위 단어 예: `~팀`, `~부서`, `~본부`, `~그룹`, `~실`, `~사업부`, `~센터(조직용)`
- 예: "LG전자 온라인기획팀" → `company="LG전자"`, `role="온라인기획팀"`

**분리하지 않는다** (중요):
- (a) 회사명 자체에 조직단위 단어가 포함된 경우. 회사명 전체를 그대로 company에 둔다.
  - 예: "LG AI 연구원" → `company="LG AI 연구원"` (연구원은 회사명 일부, 직무 아님)
  - 예: "삼성연구소" → `company="삼성연구소"`
  - 예: "현대카드" → `company="현대카드"`
- (b) 판단이 애매한 경우 → **절대 임의 분리하지 말고** 그대로 두고 사용자 확인 요청.
- (c) **company 입력에 이미 조직단위가 포함되어 있고 동시에 별도의 role이 주어진 경우** (예: "LG생활건강 디자인센터" + role "디자이너") → 내부 처리는 원본 유지(`company="LG생활건강 디자인센터"`, `role="디자이너"`)하고, 사용자에게 **"기업명 확인"** 메모로 친절하게 여쭙는다. 내부 용어("3단 구조" 등)는 노출하지 말고, LD가 자연스럽게 "디자인센터를 회사에서 빼주세요" 수준으로 답할 수 있게 유도.

**누락 방지 원칙**: 분리 시 정보 손실이 발생해서는 안 된다. 애매하면 안전하게 원본을 유지하고 사용자에게 묻는다.

**role에서 부차 정보 제거**: role 안에 "~명", "~인원", "구성원 N명" 같은 인원수 표현이 섞여 있으면 role에서는 **제거**한다. 인원수는 현재 스키마에 별도 필드로 두지 않는다 (Phase 1은 개념적 내용 설계 단계라 인원수가 직접 영향 주지 않음).

### 규칙 2: 복수 툴 처리

- `tools`는 항상 **배열**로 저장한다. 툴이 하나여도 `["ChatGPT"]`처럼 배열.
- 사용자 입력에서 `/`, `,`, `와`, `그리고` 등으로 구분된 복수 툴은 각각의 원소로 분리한다.
  - 예: "ChatGPT / Claude Code" → `["ChatGPT", "Claude Code"]`
- **"등" 표현 감지 시 강제 질문** (중요):
  - 입력에 `등`, `등등`, `기타` 같은 열린 목록 표현이 포함된 경우, 명시된 툴만 배열에 담고 **"추가 툴 확인" 메모를 답변 필수(⭐)로** 띄운다.
  - 사용자 답변을 받기 전에는 **다음 단계로 진행하지 않는다.**
  - 사용자가 "없음" / "진행" / "이대로" 등으로 명시적으로 확정하면 그때 진행한다.
- **실습 구성 방식 (고정)**:
  > 복수 툴이 입력된 경우, **분담 방식(각 툴별로 별도 실습 세트 구성)을 기본**으로 적용한다.
  > 하나의 실습 안에서 두 툴을 혼합 사용하는 "혼합 실습"은 현재 Phase에서는 **지원하지 않는다.**
  > 향후 보완 가능성이 있으며, "흐름형 실습(task 묶음 단위)" 검토 결과에 따라 Phase 2 또는 차주 보완 과제로 다룰 수 있다.

### 규칙 3: 내부 일관성 검증

`tools`, `topic`, `level` 필드들을 교차 검증하여, 같은 도구 이름이 필드마다 다르게 쓰였는지 확인한다.

**예시**:
- `tools = ["Claude"]`이지만 `level`에 "Claude Code의 경우..." 언급
- `tools = ["ChatGPT"]`이지만 `topic`에 "GPT-4o API" 언급 (ChatGPT vs GPT API는 별개)

**감지 시**: 임의로 수정하지 말고 **사용자에게 확인 요청**.
- 예: "tools에 'Claude'라고 적어주셨는데 수준 설명에서 'Claude Code'를 언급하셨어요. 두 툴은 다른데, 어느 쪽을 교육 대상으로 할까요?"

사용자가 "둘 다 모르겠다" / "그대로 두자"라고 하면 → 원본 유지하되 `notes` 필드에 불일치 감지 기록을 남긴다.

## 실행 절차

0. **초기 안내 표시**: 위의 "초기 안내" 섹션을 사용자에게 그대로 출력하고, 사용자의 답변을 기다린다.
1. 사용자 입력(대화 내용, 첨부 문서 등)에서 위 항목들을 추출한다.
2. **필수 항목 검증**: company, role, tools, topic 중 하나라도 파악되지 않으면 즉시 중단하고, 아래 "필수 누락 안내 형식"에 따라 안내한다. 이 단계에서는 **규칙 1/2/3을 적용하지 않으며**, 확인된 항목도 원본 그대로 표시한다.
3. **규칙 1 적용**: 회사/직무 분리. (필수 4개가 모두 있을 때만 이 단계 이후 진행)
4. **규칙 2 적용**: 툴을 배열로 정규화. 복수면 플래그만 세워두고 전략 결정은 후속 스킬로 미룸. "등" 감지 시 답변 필수(⭐) 메모 처리.
5. **규칙 3 적용**: 내부 일관성 검증. 불일치 발견 시 사용자 확인 요청.
6. **선택 항목 처리**: 입력에서 파악되면 해당 값을 사용하고, 파악되지 않으면 기본값을 적용.
7. 추출·처리 결과를 사용자에게 보여주고 확인을 받는다.
8. 확인을 받으면 `input.json`을 저장한다.

## 필수 누락 안내 형식

2단계에서 필수 항목 누락 시 아래 템플릿을 사용한다. 이 단계에서는 규칙 1/2/3을 적용하지 않으므로 **확인된 항목은 원본 그대로** 표시한다.

```
⚠ 필수 항목이 부족합니다

[확인된 항목]
- {필드1}: {원본값}
- {필드2}: {원본값}

[누락된 필수 항목]
- {필드 설명} (예: {구체 예시 2~3개})
- {필드 설명} (예: {구체 예시 2~3개})

※ 선택 항목({미제공된 선택 필드 목록})도 알려주시면 이후 기업 맥락에 맞춰 고도화하는 단계에서 활용됩니다 (지금 만드는 표준 커리큘럼에는 미반영).
  미입력 시 각각 기본값({매핑된 기본값 요약})으로 진행됩니다.

필수 항목을 알려주시면 이어서 진행할게요.
```

### 선택 항목 리마인드 규칙

- 선택 항목 중 **미제공이 하나라도 있을 때만** "※" 블록을 표시한다.
- 선택 항목이 모두 제공된 경우 "※" 블록은 생략한다.
- 기본값 요약은 간결하게 표기: `8시간 / "없음" / "평범한 직장인 수준"` 같은 형태.

### 필드별 예시 문구 (누락 안내용)

| 필드 | 설명 | 예시 |
|------|------|------|
| `company` | 회사명 | "LG전자", "현대카드", "삼성디스플레이" |
| `role` | 직무 | "온라인기획팀", "마케터", "디자이너" |
| `tools` | 교육에 사용할 툴 | "ChatGPT", "Claude Code", "제미나이", "미드저니" |
| `topic` | 교육 주제 | "데이터 분석 및 대시보드 구축", "영상 콘텐츠 제작", "고객 데이터 분석 자동화" |

### 선택 항목 기본값 요약 (리마인드용)

| 필드 | 기본값 요약 표기 |
|------|---------------|
| `hours` | `8시간` |
| `security` | `"없음"` |
| `level` | `"평범한 직장인 수준"` |

## 사용자 확인 형식

```
📋 입력 정보 확인

[필수]
- 회사: {company}
- 직무: {role}
- 툴: {tools as comma-separated}
- 주제: {topic}

[선택]
- 시수: {hours}시간 {미입력 시 → "(기본값)"}
- 보안 제약: {security}
- 수강생 수준: {level}

[처리 메모 — 아래 표시 조건 참조]
- ⚠ 분리 안내: "LG전자 온라인기획팀" → 회사: "LG전자" / 직무: "온라인기획팀"으로 분리했습니다. 맞나요?
- ⚠ 기업명 확인: "LG생활건강 디자인센터" 전체를 기업명으로 인식했습니다. 혹시 "디자인센터" 부분이 부서·조직명에 해당하거나 직무 정보에 합쳐야 한다면 알려주세요.
- ⚠ 복수 툴 감지: 툴이 2개(ChatGPT, Claude Code)입니다. 이후 실습 구성 단계에서 각 툴별로 별도 실습을 구성합니다.
- ⭐ 추가 툴 확인: 입력에 "등"이 포함되어 있어, 명시되지 않은 툴이 더 있을 가능성이 있습니다. 현재는 명시된 [미드저니, 제미나이, 피그마]만 사용 툴로 인식했습니다. 추가할 툴이 있으면 알려주시고, 더 없으면 "없음" 또는 "진행"이라고 답해주세요.
- ⭐ 일관성 확인: tools에 "Claude"라고 적혀있으나 수준 설명에서 "Claude Code"를 언급하셨습니다. 어느 쪽이 맞나요?
- ⚠ 제거 안내: 직무에서 "15명" 같은 인원수 표현은 제외했습니다. (Phase 1 개념 설계에 불필요)

위 내용을 확인해주세요. 수정하거나 답변할 부분이 있으면 말씀해주시고, 없으면 이대로 진행하겠습니다.
※ ⭐ 표시된 메모는 **반드시 답변이 필요**합니다. 답변 없이는 다음 단계로 진행하지 않습니다.
```

### 처리 메모 표시 조건

각 메모는 아래 조건을 충족할 때만 출력한다. 특이사항이 없으면 `[처리 메모]` 섹션 자체를 생략한다.

| 메모 | 표시 조건 | 답변 필수 |
|------|----------|----------|
| 분리 안내 | 회사/직무 분리가 실제로 일어난 경우 | - |
| 기업명 확인 | 규칙 1-(c) 적용 (company에 조직단위가 있고 별도 role도 주어진 경우) | - |
| 복수 툴 감지 | `tools.length ≥ 2` | - |
| 추가 툴 확인 | 입력에 "등", "등등", "기타" 등 열린 목록 표현이 포함된 경우 | ⭐ |
| 일관성 확인 | 규칙 3에서 불일치가 감지된 경우 | ⭐ |
| 제거 안내 | role에서 인원수 등 부차 정보를 제거한 경우 | - |

> **답변 필수(⭐) 메모가 있으면** 사용자 답변을 받기 전까지 8단계(input.json 저장)로 진행하지 않는다. 사용자가 "진행" / "OK"만 하면 "⭐ 항목에 답변이 필요합니다"라고 되물어야 한다.

## 수정 요청 처리

7단계에서 사용자가 수정 요청 시:
- 해당 항목을 업데이트하고 5~6단계로 돌아가 재검증·재확인한다.
- "OK" / "진행" / "좋아" 등으로 확인되면 8단계(input.json 저장)로 진행한다.

## Output: input.json

### 실행 폴더 생성 (Skill 1의 책임)

모든 실행은 **독립된 폴더**에 격리 저장된다. 루트는 **사용자 바탕화면 하위 `curriculum-builder-output/`** — LD가 산출물을 쉽게 찾도록 바탕화면에 둔다. Skill 1이 실행 시작 시 다음을 수행:

1. **바탕화면 경로 확인**: 실행 시점에 사용자 바탕화면 폴더의 실제 경로를 찾는다.
   - 보통 `~/Desktop` (예: `C:\Users\{user}\Desktop`). OneDrive 동기화 환경에서는 `~/OneDrive/Desktop`인 경우가 흔하다 — 둘 중 실제 존재하는 쪽을 사용.
   - 절대 경로로 확정 (예: `C:\Users\GA\Desktop\curriculum-builder-output`).
2. **폴더명 조립**: `{company}_{role}_{YYYYMMDDHHMM}`
   - 공백·특수문자는 언더스코어로 치환 (예: "LG생활건강 디자인센터" → "LG생활건강_디자인센터")
   - 타임스탬프는 실행 시각 분 단위 12자리 (예: `202604161532`)
3. **폴더 생성**: `{바탕화면}/curriculum-builder-output/{폴더명}/` + 하위 폴더 6건 자동 생성:
   - `phase1/` (Phase 1 산출 영역 — m{N}-{tool}.md + curriculum.md)
   - `phase1/task-research/` (Skill 2 산출 영역 — raw-tasks.json·atomic-tasks.json·task-dna.json·task-cards.json·workflow.json)
   - `phase2/factcheck/` (공정 1·rebuild 산출 영역 — factcheck-result.json·curriculum-post-factcheck.md·m{N}-{tool}.v2.md·validation-check-result.json)
   - `phase2/blocks/` (공정 2 산출 영역 — short.md·long.md·short-meta.json·long-meta.json)
   - `phase2/block-refinement/` (공정 3 산출 영역 — short-refined.md·long-refined.md·short-meta-refined.json·long-meta-refined.json)
   - `phase2/curriculum-final-meta/` (공정 4 부속 메타·검증 자료 영역 — final-result.json·self-check-result.json. ★ LD가 받는 최종본 md는 run_folder 직속에 저장됨, 이 폴더 아님)
4. **input.json 저장**: 해당 폴더 내부 (run_folder 직속·phase 영역 X·메타 자료)
5. **input.json에 `run_folder` 필드 포함**: **절대 경로**를 기록한다 (예: `C:\Users\GA\Desktop\curriculum-builder-output\LG전자_온라인기획팀_202604161532`). 후속 스킬은 이 절대 경로를 그대로 참조하여 같은 폴더에 저장.

**run_folder 직속 자료 영역**: `curriculum-final_{company}_{role}_{timestamp}.md` (★ LD가 받는 최종본 — 공정 4가 저장) + input.json·tasks.json·tool-features.json·top-tasks.json (Skill 1~4 메타 산출). task-research/는 `phase1/` 하위.

**저장 경로**: `{바탕화면}/curriculum-builder-output/{company}_{role}_{YYYYMMDDHHMM}/input.json`

**예시 경로**: `C:\Users\GA\Desktop\curriculum-builder-output\LG전자_온라인기획팀_202604161532\input.json`

같은 회사·직무 재실행 시 타임스탬프로 자동 구분되므로 LD 확인 없이 새 폴더 생성.

### JSON 예시

```json
{
  "company": "LG전자",
  "role": "온라인기획팀",
  "tools": ["ChatGPT", "Claude Code"],
  "topic": "데이터 분석 및 대시보드 구축",
  "hours": 8,
  "security": "외부 AI 사이트 접근 불가하여 외부 인터넷망 사용 필요, AI에 파일 업로드 안 됨, 구글 관련 사이트 접속 불가능",
  "level": "ChatGPT에 대한 기본 이해도는 있는 상황이며 효율적 활용 방안 학습 원함. Claude Code는 최근 등장한 Tool로 자동화 범위가 궁금함.",
  "run_folder": "C:\\Users\\GA\\Desktop\\curriculum-builder-output\\LG전자_온라인기획팀_202604161532",
  "notes": {
    "company_role_split": "원본 'LG전자 온라인기획팀'에서 분리함",
    "role_trim": "role에서 '구성원 15명' 인원수 표현 제거",
    "multi_tool": true,
    "consistency_check": "tools에 'Claude'가 아니라 'Claude Code'로 확정 (사용자 확인)"
  }
}
```

### 키 규약
- JSON 키는 **영문 snake_case**로 통일한다.
- 다른 서브 스킬에서도 `input.company`, `input.role`, `input.tools` 식으로 영문 키로 참조한다.
- `notes` 필드는 선택적. 특별한 처리 사항이 있을 때만 기록. 특이사항 없으면 생략 가능.

## notes 필드 읽기 규약 (후속 스킬용)

후속 스킬 구현자는 아래 규약대로 `input.notes`를 해석한다.

| 필드 | 읽어야 하는 스킬 | 처리 방식 |
|------|----------------|----------|
| `multi_tool: true` | Skill 3, 4, 5 | **Skill 3**: 각 툴별 독립 조사 (무조건). **Skill 4~5**: 분담 방식 로직 분기 (툴별로 별도 실습 세트 구성) |
| `consistency_check` | Skill 3 | 기록된 확정값 기준으로 조사 (사용자가 수정한 도구명이 있을 수 있음) |
| `company_role_split` | (참고용) | 처리 로직 영향 없음. 이력 기록 목적 |
| `role_trim` | (참고용) | 처리 로직 영향 없음. role이 왜 원본보다 짧아졌는지 역추적용 이력 |

## 금지 사항

- 필수 항목이 누락된 상태에서 임의로 추정하여 채우지 않는다.
- hours, security, level에 대해 기본값이 아닌 값을 임의로 생성하지 않는다.
- 회사/직무 분리가 애매하면 **임의로 나누지 말고** 사용자에게 묻는다.
- 복수 툴의 "실습 전략"(툴별 분담 vs 혼합)을 Skill 1에서 결정하지 않는다. Skill 4~5에서 결정할 항목이다.
- 내부 일관성 불일치를 감지해도 **임의로 수정하지 않는다**. 반드시 사용자 확인 요청.
- 툴 입력에 "등"/"등등"/"기타" 같은 열린 목록 표현이 있어도 **임의로 무시하고 통과시키지 않는다**. 반드시 답변 필수(⭐) 메모로 사용자 확인을 받은 후 진행한다.
- 이 스킬에서 커리큘럼 내용을 생성하거나 판단하지 않는다. 정보 수집과 정리만 한다.
