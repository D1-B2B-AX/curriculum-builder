# Skill 3: 툴 기능 조사

교육에 사용할 툴이 **실제로 무엇을 할 수 있는지**를 웹 검색으로 조사한다.
후속 스킬이 "이 task에 이 툴의 어떤 기능이 맞는가"를 판단할 때 근거가 되는 `tool-features.json`을 생성한다.

## 왜 이 스킬이 필요한가

Skill 4(top3 선정)에서 task × tool feature 매칭을 하려면, 그 툴이 **구체적으로 어떤 기능을 제공하는지** 목록이 먼저 있어야 한다.
또한 Skill 5~8(M4~M1 설계)에서도 "이 기능으로 이런 실습을 한다", "이 툴은 이런 도구다"를 쓸 때 이 조사 결과를 참조한다.

## Input

- `input.json`에서 **`tools`, `run_folder`**를 읽는다.
- `tools`는 배열이다 (단일 툴도 `["ChatGPT"]`). Skill 1이 보장.
- `run_folder`는 Skill 1이 생성한 실행 폴더 이름(상대 경로). 산출물은 이 폴더에 저장.

### Skill 1 notes 참조

| notes 필드 | Skill 3에서의 처리 |
|---|---|
| `multi_tool: true` | 각 툴별 독립 조사 실행 (무조건) |
| `consistency_check` | 기록된 확정값 기준으로 조사. 예: notes에 "tools에 'Claude'가 아니라 'Claude Code'로 확정"이라 되어 있으면 Claude Code로 조사 |

### 사용하지 않는 필드

`company`, `role`, `topic`, `hours`, `security`, `level`은 이 스킬에서 사용하지 않는다.
이 스킬은 **툴 자체의 기능**을 조사하는 단계이지, 특정 회사·직무·주제에 맞추는 단계가 아니다.
"이 직무에 어떤 기능이 적합한가"는 Skill 4의 역할이다.

---

## 정보 소스

**웹 검색 단독** (2026-04-14 세션 결정).

이유:
- LLM에 참고 자료(스킬팩 등)를 주면 그 자료에만 의존해 판단할 위험이 있음 (anchoring)
- 웹 검색이 최신 기능 반영에 유리함 (툴은 수시로 업데이트됨)
- 테스트 후 결과가 부족하면 추후 참고 자료 추가 여부 재결정

## 웹 검색 전략

### 검색 대상 (우선순위순)

각 툴에 대해 아래 소스를 순서대로 탐색한다. 모든 소스를 반드시 다 찾을 필요는 없지만, **최소 1·2번은 반드시 시도**한다.

| 순위 | 소스 | 목적 | 예시 검색어 |
|------|------|------|-----------|
| 1 | **공식 사이트/문서** | 기능 목록의 1차 출처 | `{tool_name} features site:{공식도메인}` |
| 2 | **최근 업데이트/릴리즈 노트** | 신규·변경 기능 반영 | `{tool_name} new features 2026`, `{tool_name} release notes` |
| 3 | **주요 리뷰/비교 기사** | 공식 문서에서 놓친 기능·관점 보완 | `{tool_name} review features 2026`, `{tool_name} vs {경쟁툴}` |

### 검색 시 주의

- **버전/플랜 구분**: 같은 툴이라도 무료/유료, 개인/기업 버전에 따라 기능이 다를 수 있다. 확인 가능한 범위 내에서 버전/플랜 정보를 `version_info`에 기록한다. 확인 불가면 "일반 사용자 기준"으로 조사.
- **beta/실험 기능**: 공식 릴리즈된 기능과 beta/실험 기능을 구분한다. beta는 feature에 `"(beta)"` 표기.
- **중복 기능명 주의**: 같은 기능이 시기에 따라 다른 이름으로 불리는 경우(예: "Code Interpreter" → "Advanced Data Analysis") 가장 최신 명칭을 `name`에, 구명칭은 `description`에 언급.

## 실행 절차

1. `input.json`을 읽어 `tools`, `run_folder`를 확인한다. `notes`에 `consistency_check` 기록이 있으면 해당 확정값을 기준으로 삼는다.
2. LD에게 시작 안내를 간결하게 표시한다:

   > "Skill 3 (툴 기능 조사)을 시작합니다. {tools 목록}의 주요 기능을 웹 검색으로 조사합니다."

3. **각 툴별 독립 조사**를 실행한다. `tools` 배열의 각 원소에 대해:
   a. 위 "웹 검색 전략"에 따라 검색 실행
   b. 검색 결과에서 기능을 추출하여 features 배열 구성
   c. 조사 커버리지 자체 점검 실행 (아래 섹션 참조)
   d. 자체 점검에서 보완 필요 판정 시, 추가 검색 1회 실행

4. 전체 결과를 `tool-features.json`으로 저장한다.
5. LD에게 조사 결과 요약을 보고한다:

```
✅ 툴 기능 조사 완료

{tool_name_1}: 주요 기능 {N}개 파악
  • {feature_name_1} — {한줄 설명}
  • {feature_name_2} — {한줄 설명}
  • ...

{tool_name_2 (복수 툴인 경우)}: 주요 기능 {M}개 파악
  • ...

※ 웹 검색 기반 조사이므로 미포함된 기능이 있을 수 있습니다. 빠진 기능이 있으면 알려주세요.

산출물: curriculum-builder-output/{run_folder}/tool-features.json

확인 후 다음 단계(실습 task 선정)로 넘어가겠습니다.
```

---

## 조사 커버리지 자체 점검

기능 개수의 하한은 두지 않는다. 특화 도구(예: Midjourney)는 핵심 기능이 3~4개일 수 있으며, 그것이 조사 실패를 의미하지 않는다.

대신 아래 **조사 범위** 기준으로 자체 점검한다:

| 점검 항목 | 기준 | 미충족 시 |
|----------|------|----------|
| 공식 문서 확인 | 공식 사이트 또는 공식 문서를 최소 1건 이상 참조했는가 | 추가 검색 1회 시도 |
| 최신성 | 조사한 정보가 1년 이내(2025년 이후) 기준인가 | 릴리즈 노트 검색 추가 |
| 핵심 기능 커버 | 해당 툴을 소개할 때 빠질 수 없는 대표 기능이 포함되어 있는가 (LLM 자체 판단) | 추가 검색 1회 시도 |

자체 점검은 LD에게 노출하지 않는다. 보완 검색 후에도 미충족이면 **그 상태로 저장**하되, `completeness_note`에 어떤 점검이 미충족인지 기록한다.

### 필드 규약

**툴 단위 (각 키의 값)**:

| 필드 | 타입 | 설명 |
|------|------|------|
| (키 자체) | string | input.json의 tools 배열 원소와 동일한 툴명 |
| `version_info` | string | 조사 기준이 된 버전/플랜/시점. 확인 불가 시 "일반 사용자 기준, {날짜} 시점" |
| `features` | array | 해당 툴의 주요 기능 목록 |
| `sources` | string[] | 조사에 사용한 출처 (공식 문서, 리뷰 등). URL 또는 출처 설명 |
| `survey_date` | string | 조사 실행 날짜 (YYYY-MM-DD) |
| `completeness_note` | string | 조사 범위 경고. 고정값: "주요 기능 중심 조사. 미기재 기능 있을 수 있음." 자체 점검 미충족 시 사유 추가 |

**기능 단위 (`features[]`)**:

| 필드 | 타입 | 설명 |
|------|------|------|
| `name` | string | 기능의 공식 명칭 (최신 명칭 사용, 구명칭은 description에 병기) |
| `description` | string | 이 기능이 **구체적으로** 무엇을 하는지 서술. 후속 스킬이 인용(quote)할 수 있을 만큼 명확해야 한다 — 아래 "description 작성 규칙" 참조 |
| `useful_for` | string | 어떤 유형의 업무에 유용한지 (1~2문장) |
| `caveats` | string \| null | 베타/유료 플랜 한정/지역 제한 등 주의사항. 해당 없으면 `null`. Skill 8(M1 소개)에서 활용 가능 |

### description 작성 규칙

`description`은 후속 스킬(특히 Skill 4)이 task의 행위와 매칭할 때 **인용 근거**로 사용된다. 다음 규칙을 따른다:

- **구체 동작 중심으로 서술**: 이 기능으로 **무엇을 넣으면 → 무엇을 하고 → 무엇이 나오는지**가 드러나야 한다.
  - 좋은 예: "CSV/Excel 파일을 업로드하면 Python 코드를 자동 생성·실행하여 데이터를 분석하고 차트를 생성한다."
  - 좋은 예: "긴 문서·코드를 전용 편집 패널에서 인라인 편집하고, 버전 비교·섹션별 피드백 반영이 가능하다."
- **추상 표현 금지**: "텍스트 작성", "다양한 분석 가능", "강력한 기능" 같은 모호한 표현은 쓰지 않는다. 이런 표현은 Skill 4 매칭 단계에서 무력화된다 (어떤 task에든 매칭되므로 변별력 없음).
- **한 기능의 세부 옵션을 별개 기능으로 쪼개지 않는다**: 예: "차트 생성"과 "그래프 생성"을 별개로 두지 않음. description 안에서 세부 옵션을 나열하는 것은 OK.

### 기능 개수 가이드

기능 개수의 **하한은 두지 않는다.** 특화 도구(예: Midjourney)는 핵심 기능이 3~4개일 수 있으며, 그것이 조사 실패를 의미하지 않는다.

**상한 가이드**: 한 툴당 **15개 이하**를 권장한다. 범용 도구(ChatGPT 등)에서 세부 옵션까지 나열하면 20개 이상 나올 수 있는데, 후속 스킬(Skill 4)의 LLM 평가 부담이 커지므로 핵심 기능 위주로 정리한다. 세부 옵션은 description 안에서 언급하는 것으로 충분.

## 실패 처리

### 자동 재시도 (LD에게 알리지 않음)

웹 검색 결과가 비어있거나 파싱 실패 시, 검색어를 변경하여 최대 2회 자동 재시도.
- 1차 재시도: 검색어 단순화 (예: 공식 도메인 제한 해제)
- 2차 재시도: 영문 검색어로 변경 (한글 툴명→영문 툴명)

### 조사 결과가 극단적으로 부족한 경우

**조건**: 특정 툴에 대해 재시도 후에도 features를 1개도 도출하지 못한 경우

**템플릿**:

```
⚠ {tool_name}에 대한 기능 정보를 찾지 못했습니다

웹 검색으로 {tool_name}의 기능 정보를 충분히 확보하지 못했습니다.
다음 중 하나를 선택해주세요:

1. 이 툴에 대해 알고 계신 기능을 직접 알려주시면 반영하겠습니다
2. 다른 툴명이나 정확한 명칭이 있다면 알려주세요
3. 이 툴을 빼고 나머지 툴로 진행합니다
```

### 복수 툴 중 일부만 실패한 경우

성공한 툴의 결과는 정상 저장하고, 실패한 툴에 대해서만 위 안내를 표시한다.
LD가 실패 툴을 제외하기로 하면, `tool-features.json`에서 해당 툴의 `features`를 빈 배열로 두고 `completeness_note`에 사유를 기록한다.

---

## LD 피드백 처리

조사 결과 요약 후 LD의 답변에 따라 다음과 같이 대응한다:

- **"진행해" / "OK"** → Skill 4로 진행
- **"이 기능이 빠졌다"** → LD가 알려준 기능을 features에 추가. `sources`에 "LD 직접 제공"으로 기록. tool-features.json 재저장
- **"이 기능은 없어졌다 / 틀렸다"** → 해당 feature 제거 또는 수정. tool-features.json 재저장
- **"다시 조사해줘"** → 해당 툴에 대해 웹 검색 재실행

Skill 3이 **임의로 결과를 수정하지 않는다.** LD의 의도를 확인하고 반영한다.

## 금지 사항

- **LLM 지식만으로 features를 채우지 않는다.** 반드시 웹 검색을 실행하고, 검색 결과에 근거하여 작성한다. LLM이 "알고 있는" 기능이라도 웹 검색으로 확인되지 않으면 넣지 않는다.
- **기능을 과장하거나 축소하지 않는다.** 검색 결과에 나온 그대로 서술한다.
- **특정 직무·주제에 맞춰 기능을 선별하지 않는다.** 이 스킬은 툴 자체의 기능을 조사하는 단계다. "마케터에게 유용한 기능만 조사" 같은 필터링은 하지 않는다. 직무 맥락에서의 선별은 Skill 4의 역할.
- **기능 개수를 인위적으로 맞추지 않는다.** 특화 도구라서 기능이 3개면 3개로 둔다. 범용 도구에서 15개를 넘으면 핵심 기능 위주로 정리하되, 하한을 채우기 위해 억지로 늘리지 않는다.
- **description을 추상적으로 쓰지 않는다.** "텍스트 작성", "데이터 분석 지원", "다양한 기능 제공" 같은 표현은 Skill 4 매칭 단계에서 변별력이 없어 무력화된다. 반드시 구체 동작(입력→처리→출력)이 드러나게 쓴다.
