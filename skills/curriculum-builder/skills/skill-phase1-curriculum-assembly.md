# Phase 1 조합: Curriculum Assembly (Phase 1 순수)

## 역할

Skill 5·6·7·8이 만든 툴별 m1·m2·m3·m4 모듈 구성 기획 문서를 읽어 **단일 `curriculum.md`**로 조합한다.

구조:
- **메인**: 툴별 커리큘럼 표 (제안서 삽입·강사 공유용)
- **보충**: 표 아래 "모듈 구성 설명" — 왜 이렇게 구성되었는지 LD 검토용

복수 툴이면 한 파일 안에 툴별 섹션 나란히 배치. LD는 이를 하나의 통합 과정으로 묶을지 별도 과정으로 운영할지 판단. 산출물은 중립.

## Phase 1 순수 원칙

- **시수 미포함** — curriculum.md에 시수를 박지 않는다. 시수 정리는 Phase 2 공정 2(hours-blocks)가 표준 grid(short 6h·long 12h)로 담당
- **보안·수준 미반영** — `input.json.security/level`은 산출물 본문에 박지 X. 운영 제약 반영은 Phase 3 자유 대화 영역
- **m1~m4 원문의 성격 유지** — 모듈 구성 기획 수준 (구체 발화·강의 스크립트·슬라이드 대본 박지 말 것·학습 주제·메인 학습 내용은 OK)
- **원문에 없는 내용 창작 금지** — m1~m4에 명시된 꼭지·내용만 사용

## 실행 순서

본 스킬은 **Phase 1의 마지막 단계**다. 흐름: `M1~M4 (skill 5·6·7·8) → Phase 1 조합(이 스킬·curriculum.md 산출) → Phase 2 공정 1(factcheck) → 공정 1 후속(curriculum-rebuild·curriculum-post-factcheck.md) → 공정 2(hours-blocks) → 공정 3(block-refinement) → 공정 4(curriculum-final·LD가 받는 최종본)`.

- `curriculum.md` = Phase 1 조합의 정식 산출물 (시수·검증 전 단계의 중간본). Phase 2 공정 1 후속(rebuild)이 이를 받아 `curriculum-post-factcheck.md`로 넘긴다.
- 시수·보안·수준 같은 운영 제약은 Phase 2까지의 표준 커리큘럼에 반영되지 않으며(Phase 2는 표준 grid 고정), Phase 3 LD-Claude 자유 대화에서 반영된다.
- m1~m4 중 하나라도 수정되면 본 스킬을 재실행해 curriculum.md를 다시 조립한다 (→ 그 영향으로 Phase 2 공정들도 연쇄 재실행).

### Bullet 작성 원칙 (커리큘럼 표) — 매우 중요

**커리큘럼 표 bullet의 본질**:
> 수강생 입장에서 **"이 모듈에서 무엇을 배우는가 / 어떤 실습을 하는가"의 학습 주제·실습 단위**를 한 줄로 요약한 것.
>
> 모듈 내용을 통째로 옮기거나 방식·흐름을 서술하는 것이 **아님**.

**변환 원칙**:

1. **꼭지 제목이 주제형·구체적이면 → 그대로 bullet**
   - 예: "주요 기능 카테고리" → bullet 그대로 "ChatGPT 주요 기능 카테고리"

2. **꼭지 제목이 설명형·의문형이면 → 주제로 변환**
   - 예: "이 task가 왜 AI를 필요로 하는가" → ❌ 그대로 옮기면 부자연
   - → ✅ "AI 전환 가치 · 수작업 한계" 등 **학습 주제로 변환**

3. **꼭지 내용 통째로 옮기기 금지**
   - ❌ "ChatGPT는 OpenAI의 대화형 범용 AI — 텍스트 생성·데이터 분석... 올인원 도구" (내용 통째)
   - ✅ "OpenAI ChatGPT의 특징" 또는 "대화형 범용 AI의 기본 성격" (학습 주제)

4. **설계 의도·LD 검토용 서술어 금지** (커리큘럼 표는 수강생 관점)
   - ❌ "업무 흐름 내 위치", "실무에서 만나는 전형 상황", "이 task를 왜 선정했는가" (LD 검토용 표현)
   - ✅ 이런 내용은 **"모듈 구성 설명"** 섹션으로 이동. 표 bullet에는 **수강생이 배우는 주제**만

4-1. **★ "LD·LD 리뷰용·LD 검토용" 같은 회사 내부 워딩 표 bullet 박지 X 금지**
   - curriculum.md는 외부 고객사·강사 공유용 자료. LD(Learning Designer)는 우리 회사 내부 호칭이라 외부 노출 X
   - ❌ "컨셉 후보 시안 보드 정돈 (LD 리뷰용)", "산출물 정리 (LD 검토용)" 같은 표 bullet
   - ✅ "컨셉 후보 시안 보드 정돈" — 학습 주제·실습 단위만 표에 / LD 검토 의도는 모듈 구성 설명 섹션에 자연 녹임

4-2. **★ 수강 흐름 안내 표 bullet 박지 X 금지**
   - M1~M4 연결·M2 → M3 → M4 학습 흐름 안내는 메타 영역, 실제 학습 내용 X
   - 표는 수강생이 배우는 학습 주제·실습 단위만. 학습 흐름 자체는 별도 안내 영역 (SKILL.md 파일 구조의 § "수강 흐름" 헤더·각 모듈 .md의 §"M3 · M4 학습 흐름 연결" 섹션에 이미 박힘)
   - ❌ "M3 기초 → M4 실전 학습 흐름 안내" 같은 표 bullet
   - skill-7(M2)·skill-8(M1) spec도 동일 정신 (수강 흐름 안내 꼭지 금지)

5. **M4 실습 특수 규정**:
   - ❌ Step 1~N을 그대로 bullet로 나열 금지 (step은 실습 내 흐름일 뿐)
   - ✅ Step들을 **통합된 실습 단위 3~4개**로 요약
   - 예: 5 Step(데이터 준비 → 키워드 탐색 → 분류 → 인사이트 → 보고서) → **"VOC 데이터 ADA 분석 / 개선 인사이트 도출 / 경영진 보고서 작성"** (3개 실습 단위)

6. **각 bullet 1줄** (길면 2줄까지). 학습 주제·실습 단위는 원래 짧게 표현 가능

7. **M1~M4 각각 3~5개 bullet** 권장

8. **시수·보안·수준 언급 금지** (Phase 1 원칙)

9. **참고 구조 — BGF리테일 제안서 커리큘럼 표**:
   BGF 제안서 36p 표는 `일자 | 주제 | 주요 내용 | 시수` 구조. 각 행:
   - **주제** = 간결한 명사구 한 줄 (예: "비정형 데이터 분석", "VBA 기반 업무 자동화")
   - **주요 내용** = 그 주제 아래 이론·기법·실습 내용을 아우르는 bullet 여러 개 (+ BGF는 [실습] 마커로 실습 블록 명시)
   
   우리 Phase 1은 `일자`·`시수` 생략, `구분 | 내용` 구조 유지:
   - 구분 열의 "M1 (...)" = BGF의 "주제"
   - 내용 열의 bullet들 = BGF의 "주요 내용"
   - **M3·M4는 모듈 자체가 실습이라 `[실습]` 마커 불필요**. 실습 내용이 잘 드러나기만 하면 됨

10. **M3 특수 규정 — 독립성 기준 유연 배치**:
    - bullet 개수 강제 없음 (1개·여러 개 모두 OK). 판단 기준은 "각 bullet이 독립된 실습 주제로 성립하는가"
    - 세부 활동이 독립된 실습 주제로 성립 → **여러 bullet** (예: "CSV 업로드·Python 자동 분석"처럼 구체 활동)
    - 세부가 방식·방법만 남아 독립성 부족 → **전체를 아우르는 포괄 주제 1 bullet** (예: "VOC 분석에 쓰이는 ChatGPT 핵심 기능 사용법 다지기")
    - ❌ 금지: 기능명 반복("XX 기초"), 활동 방식만 남는 bullet("단계별 추론 과정 관찰"·"연속 대화로 텍스트 점진 개선")
    - M4와의 연관성이 드러나면 더 자연스러움 (왜 이 기초인지 파악 가능)

11. **M4 특수 규정 — 실습 내용이 잘 드러나게**:
    - **실습 단위 3~4개**로 구성. 각 실습 단위에 유형(어떤 AI 활동인지)과 주제(무엇을 다루는지)가 모두 드러나야 함
    - Step 1~N 나열 금지. 통합된 실습 단위로 압축
    - `[실습]` 태그 불필요 — M4 자체가 실습이므로 실습 내용이 bullet 자체로 드러나기만 하면 됨

### 모듈 구성 설명 작성 원칙 — 중요

**역할 분리 — 표와의 관계**:
- **표 bullet** = 수강생이 배우는 **학습 주제·실습 단위** (간결, 정돈)
- **모듈 구성 설명** = "왜 이 주제들이 선정되었는지" + **"어떻게 배우는지"** + **구체적 내용 간략 설명** (표에서 빠진 맥락을 여기서 채움)

즉 표에 넣기 부자연스러운 "방식·흐름·설계 의도"는 모두 **이 섹션에서 자연스럽게 서술**.

**작성 원칙**:

1. **m1~m4 원문을 참고**해서 "왜 이 모듈이 이렇게 구성되었는지 + 어떻게 배우는지 + 구체적 내용"을 통합해 서술
2. **3~5줄 분량** — 너무 길면 표 중심이 흐려짐. 그래도 "어떻게·무엇을" 충분히 드러나야 함
3. **자연스러운 산문체** — bullet로 잘라내지 않고 문단으로. 단 3~5줄 이내
4. **원문에 없는 내용 창작 금지** — 풀어 설명은 OK, 새 내용은 X
5. 각 모듈별 중점:
   - **M1 설명**: 도구 유형·기본 사용 방식·기능 카테고리를 어떻게 소개하는지, 왜 이 구성인지
   - **M2 설명**: When(업무 맥락)·Why(적합성) 근거가 어떻게 수강생 납득으로 이어지는지 (V3·V4·V2 근거 포함)
   - **M3 설명**: M4에 필요한 feature를 왜 이렇게 골랐고 어떤 방식으로 익히는지 (역산 관점)
   - **M4 설명**: 선정 task와 실습이 실제로 어떤 흐름으로 진행되는지, 수강생이 어떤 산출물을 얻는지 (Step 흐름을 여기서 서술)
6. **구체 발화·강의 스크립트·슬라이드 대본 박지 말 것** — 기획 수준 유지 (학습 주제·메인 학습 내용은 OK)

### 복수 툴 처리
- 한 파일 안에 툴별 `## [{tool_name}]` 헤더로 섹션 나란히

## 실행 절차 (4 Step)

### Step 1. 파일 로드

`{run_folder}/phase1/`에서:
- 각 툴별 `m1-{tool}.md`, `m2-{tool}.md`, `m3-{tool}.md`, `m4-{tool}.md` 로드
- `input.json` 로드

**툴별 순회** — 각 툴에 대해 Step 2 반복.

### Step 2. 툴별 "표 + 모듈 구성 설명" 생성 (LLM)

**프롬프트**:

```
당신은 {tool_name} 커리큘럼의 **표 + 모듈 구성 설명**을 작성합니다.

[Input]
- m1-{tool}.md 전문
- m2-{tool}.md 전문
- m3-{tool}.md 전문
- m4-{tool}.md 전문
- input.topic: {topic}

[Output 구조 — 순서대로]

1. 커리큘럼 표 (4행)
   | M1 ({tool_name} 소개) | bullet들 |
   | M2 (활용 맥락) | bullet들 |
   | M3 (기초 실습) | bullet들 |
   | M4 (메인 실습) | bullet들 |

2. 모듈 구성 설명 (LD 검토용) — 각 모듈 3~4줄씩
   - M1: 도구 유형·기본 사용 방식·기능 개괄이라는 세 꼭지가 어떻게 묶이는지
   - M2: 왜 이 task가 선정되었고 왜 이 툴이 적합한가의 핵심 (V3·V4·V2 요약 포함)
   - M3: 어떤 feature들을 왜 다루는지, M4 역산으로 왜 이 구성인지
   - M4: 수강생이 실제로 무엇을 수행하고 어떤 산출물을 얻는지 (선정 근거 포함)

[Bullet 작성 원칙 — 표 작성 시 / 매우 중요]

**표 bullet의 본질**: 수강생 입장에서 "이 모듈에서 무엇을 배우는가·어떤 실습을 하는가"의 **학습 주제·실습 단위**를 한 줄로 요약. 모듈 내용 통째로 옮기거나 설계 의도·흐름 서술이 아님.

1. **꼭지 제목 변환 규칙**:
   - 주제형·구체적 제목 → 그대로 bullet OK
   - 설명형·의문형 제목 → **학습 주제로 변환** (예: "이 task가 왜 AI를 필요로 하는가" → "AI 전환 가치·수작업 한계")

2. **꼭지 내용 통째로 옮기기 금지** — 핵심 학습 주제만 추출

3. **설계 의도·LD 검토용 서술어 금지** — "업무 흐름 내 위치", "실무에서 만나는 전형 상황", "이 task를 왜 선정했는가" 등은 표에서 빼고 "모듈 구성 설명" 섹션으로 이동

3-1. **★ "LD·LD 리뷰용·LD 검토용" 같은 회사 내부 워딩 표 bullet 박지 X** — curriculum.md는 외부 고객사·강사 공유 자료. LD(Learning Designer)는 회사 내부 호칭. 표 bullet에 "(LD 리뷰용)·(LD 검토용)" 박지 말 것. LD 검토 의도는 모듈 구성 설명 섹션에 자연 녹임

3-2. **★ 수강 흐름 안내 표 bullet 박지 X** — M1~M4 연결·M2 → M3 → M4 학습 흐름 안내는 메타 영역, 실제 학습 내용 X. 표는 수강생이 배우는 학습 주제·실습 단위만. ❌ "M3 기초 → M4 실전 학습 흐름 안내" 같은 표 bullet 박지 말 것

4. **M4 특수**: Step 1~N을 그대로 bullet로 나열 금지. **통합된 실습 단위 3~4개**로 요약 (예: 5 Step → "VOC ADA 분석 · 인사이트 도출 · 보고서 작성")

5. 각 bullet 1줄 (길면 2줄), M1~M4 각각 3~5개 권장
6. 시수·보안·수준 언급 금지

[참고 — Before/After 예시]

| 모듈 | ❌ 잘못 | ✅ 올바름 |
|---|---|---|
| M1 | "ChatGPT는 OpenAI의 대화형 범용 AI — 텍스트·데이터·이미지 등을 처리하는 올인원 도구" | "OpenAI ChatGPT의 정체와 도구 유형" |
| M2 | "이 task가 왜 AI를 필요로 하는가 — 대량 VOC 전수 분류는 수작업 불가..." | "온라인기획에 유용한 ChatGPT 기능 · AI 전환 가치" |
| M4 | Step 1~5 나열 | "VOC 데이터 ADA 분석 · 개선 인사이트 도출 · 경영진 보고서 작성" |

[모듈 구성 설명 작성 원칙 — 표와 역할 분리]

**표는 학습 주제만, 설명은 "왜·어떻게·구체적 내용"**.

표에 넣기 부자연스러운 항목(설계 의도, 배우는 방식·흐름, Step 세부, 업무 맥락 서술 등)은 모두 **이 설명 섹션에서 자연스럽게 서술**. 표에서 빠진 맥락을 여기서 채운다.

1. 각 모듈별 3~5줄 자연스러운 산문체 (bullet 금지)
2. m1~m4 원문 참고해서 "왜 이 구성인지 + 어떻게 배우는지 + 구체적 내용 간략 설명" 통합
3. 원문에 없는 내용 창작 금지. 풀어 설명은 OK, 새 내용은 X
4. 각 모듈별 중점:
   - M1: 도구 유형·사용 방식·기능 카테고리를 어떻게 소개하는지
   - M2: When·Why (V3·V4·V2 근거) 가 어떻게 납득으로 이어지는지
   - M3: M4 역산으로 feature 선정·학습 방식 (무엇을 어떻게)
   - M4: 선정 task와 실습 흐름 (Step을 여기서 서술) + 수강생 산출물
5. 구체 발화·강의 스크립트·슬라이드 대본 박지 말 것·기획 수준 유지 (학습 주제·메인 학습 내용은 OK)
6. 시수·보안·수준 언급 금지 (Phase 1 원칙)

[출력 형식]
SKILL.md §"파일 구조"의 [tool_name] 섹션 템플릿 그대로 (커리큘럼 표 + 모듈 구성 설명 순서).
```

### Step 3. 통합 파일 조립 및 저장

- 툴별 섹션(Step 2 결과) 나란히

→ `{run_folder}/phase1/curriculum.md`로 저장.

### 자동 재시도
LLM 호출 실패 시 최대 2회 재시도.

### Case A (LD 조치 가능)
- m1~m4 중 일부 누락 → 해당 Skill 재실행 유도:

## 금지 사항

- **원문에 없는 내용 창작 금지** — m1~m4 원문의 꼭지·내용만 사용. 풀어 설명은 OK, 새 내용 창작은 X
- **구체 발화·강의 스크립트·슬라이드 대본 박지 말 것** — 모듈 구성 기획 수준 유지 (학습 주제·메인 학습 내용은 OK)
- **꼭지 제목만 기계적 복사 금지** (표 bullet) — 내용 누락 위험
- **시수·분량 수치 기입 금지** — "N시간", "N분" 등. Phase 2 영역
- **보안·수준 관련 언급 금지** — Phase 1 원칙
- **복수 툴을 하나의 표·섹션으로 병합 금지** — 툴별 섹션·표 분리 유지 (LD 판단 여지 남김)
- **★ 시수·수강생 수준·보안 제약 영역 산출물 박지 X 금지** — Phase 3 영역 (input.json.security/hours/level 영역 curriculum.md 본문에 박지 말 것). LD가 input.json 자체 영역에서 별도 검토
- **★ 회사·직무 자연 녹이기 경계선 — Phase 3 영역 위반 금지** — company 이름·role 명칭·topic 키워드 자연 녹이기는 OK / 회사 사업 전략·매출·시장 점유율·조직 구조는 Phase 3 영역. ★ 직무 구체성 조건부 — input.json.role이 세부 직무 명시면 그 기준 그대로 / 포괄 직무[디자이너·마케터·개발자·기획자·영업·운영 등]면 그 직무 안의 다양한 세부 영역 자연 녹이기·세부 직무 1개로 임의 좁히기 금지
- **★ "LD·LD 리뷰용·LD 검토용" 같은 회사 내부 워딩 표 bullet 박지 X 금지** — curriculum.md는 외부 고객사·강사 공유용 자료. LD(Learning Designer)는 우리 회사 내부 호칭이라 외부 노출 영역 X. 표 bullet에 "(LD 리뷰용)·(LD 검토용)" 박지 말 것. LD 검토 의도는 모듈 구성 설명 섹션에 자연 녹임
- **★ 수강 흐름 안내 표 bullet 박지 X 금지** — M1~M4 연결·M2 → M3 → M4 학습 흐름 안내는 메타 영역, 실제 학습 내용 X. 표는 수강생이 배우는 학습 주제·실습 단위만. 학습 흐름 자체는 SKILL.md 파일 구조의 § "수강 흐름" 헤더·각 모듈 .md §"M3 · M4 학습 흐름 연결" 섹션에 이미 박힘. skill-7(M2)·skill-8(M1) spec도 동일 정신
- **두 개 파일(detail + table)로 분리 금지** — 단일 `curriculum.md`로 유지. 표(메인) + 모듈 구성 설명(보충) 통합 구조
