# Skill 8: M1 툴 소개 모듈 구성 (Phase 1 순수)

## 역할

Skill 3이 조사한 툴 기능 정보를 바탕으로, 수강생이 **"이 툴이 무엇이고 어떤 일을 할 수 있는지"**를 처음 이해하도록 돕는 M1 도입 모듈을 설계한다.

**수강 흐름상 위치**: **M1(툴 소개)** → M2(When/Why) → M3(기초) → M4(메인 실습). M1은 가장 먼저 만나는 도입부.

### M1의 공식 정의 (설계도 §1)
- **What = 사용하고자 하는 툴**
- 이 툴이 **무엇인지** 소개 — 단, 단순한 기능 나열이 아니라 **"이 AI가 어떤 도구이고 기본적으로 어떻게 사용하는지"** 이론·기본 감에 집중
- M2(왜/When·Why)·M3(기능별 기초 사용법)·M4(실전)와 명확히 분리: M1은 **task-agnostic**한 도구 소개·이론 도입부

### M1의 콘텐츠 성격 — 매우 중요
- **구체 발화·강의 스크립트·슬라이드 대본 박지 말 것** (학습 주제·메인 학습 내용은 OK)
- **모듈 구성 기획 수준**: LD가 "이 M1 모듈이 이런 꼭지들로 구성된다"를 한눈에 파악
- Skill 5(M4)·Skill 6(M3)·Skill 7(M2)과 동일한 성격

## Phase 1 순수 원칙

- **보안·시수·수준 미반영** (`input.json.security/hours/level` 읽지 않음 — Phase 3 영역)
- **feature_caveats는 LD 인지용 "※ 참고" 표시만** (대응 설계 X, Phase 3 영역)
- **구체 시간·분량 수치 금지** ("N분 강의" 등)

---

## Input

| 파일 | 읽는 필드 | 용도 |
|---|---|---|
| `tool-features.json` (Skill 3 산출) | 해당 툴 블록의 version_info / features[] (name, description, useful_for, caveats) / sources / survey_date / completeness_note | M1의 핵심 소스 |
| `input.json` | `tools[]`, `topic`, `company`, `role` — **security·hours·level 미사용** | 주제 맥락 반영 |
| `top-tasks.json` (Skill 4 산출, 있으면 참조) | `topic_input` (is_composite, topic_elements) / matched_topic | 복수 주제일 때 주제 맥락 일관성 |

### 사용하지 않는 필드
- `input.json.security/hours/level` — Phase 3 영역

### top-tasks.json 참조 여부
- 있으면 `topic_input`·`matched_topic`으로 주제 맥락 반영
- 없으면 (M1이 M2·M3·M4보다 먼저 실행되는 경우) `input.json.topic`만 사용. 복수 주제 감지는 내부 LLM으로 간단히 수행 가능 (prompts_meta.md 같은 별도 프롬프트 없이)

## Output: m1-{tool}.md

**저장 경로**: `{run_folder}/phase1/m1-{tool_name_safe}.md`

파일명 규칙은 Skill 5·6·7과 동일 (공백·특수문자 언더스코어 치환, 툴별 1개).

### 파일 구조

```markdown
# M1 (What) — {tool_name} 소개 (주제: {topic})

> 생성 정보
> - Skill 8 (M1 설계) — curriculum-builder
> - 생성일: {YYYY-MM-DD}
> - 대상 툴: {tool_name}
> - 주제: {input.topic}
> - Phase 1 순수 설계 (보안·시수·수준 미반영 — Phase 3 자유 대화에서 반영)
> - **수강 흐름**: **M1(툴 소개)** → M2(When/Why) → M3(기초) → M4(메인 실습)

## 학습 목표

이 M1 모듈을 마치면 수강생은:
- **{tool_name}**이 어떤 도구인지 한 문장으로 설명할 수 있다
- 이 툴이 주로 잘 하는 일과 한계(베타·유료 플랜 등)를 구분해서 이해한다
- 주제 "{topic}" 맥락에서 이 툴이 왜 등장하는지 맥락을 잡는다

## 툴 기본 정보 (Skill 3 조사 기반)

- **버전**: {version_info}
- **조사일**: {survey_date}
- **핵심 기능 수**: {features 개수}개 (파악된 주요 기능 기준)

## 강의 구성 제안 (M1 모듈의 주 본문 — 모듈 구성 기획 수준)

> 본 산출은 LD가 모듈 구성을 한눈에 파악하는 **기획 명세 수준**.
> 구체 정의·구체 사용 원리·구체 설명·발화 스크립트는 **강사 교안 영역** (박지 않음).
> 본 도구는 "표준 모듈 원형" 제공 (각 툴별 별도 산출 — 1 모듈 = 1 도구 중심).

### bullet 작성 패턴 (제안서 표준 source 정합)

- **학습 동사** (이해·정리·판단·구분·도출·비교 등) + **영역 명세** + 콜론·괄호 뒤 **1-2개 카테고리 자연 녹이기**
- 자연 녹일 수 있는 영역:
  - 도구 자체 속성 (도구 유형 일반 카테고리·도구 기능·인터랙션 패턴)
  - input.json source (company 이름·role 명칭·topic 키워드 OK / 회사 사업 전략·매출·시장 점유율·조직 구조는 Phase 3 영역. ★ 직무 구체성 조건부 — input.json.role이 세부 직무 명시면 그 기준 그대로 / 포괄 직무[디자이너·마케터·개발자·기획자·영업·운영 등]면 그 직무 안의 다양한 세부 영역 자연 녹이기·세부 직무 1개로 임의 좁히기 금지)
- 박지 말 것:
  - input.json.tools 내 다른 도구 비교 (각 도구별 별도 산출이라 모듈 영역 침범)
  - 구체 타사 제품명 거론 (input.json에 없는 도구 — hallucination)
  - 회사 사업 맥락 (Phase 3 영역)
  - 임의 통계·인용·도입 사례
  - 풀어 쓴 설명문·발화 스크립트·강사 워딩

### 필수 포함 학습 영역 (어떤 꼭지 형태·꼭지 수로든 반드시 다룰 것)

1. **도구 정체** — 이 AI가 어떤 종류의 도구인지 인식·정리하는 학습 영역
2. **기본 사용 원리** — 이 도구의 사용 패턴·인터랙션 인식·이해하는 학습 영역
3. **주요 기능 개괄** — 핵심 기능 카테고리 인식 (세부 사용법은 M3 영역)

### 금지 영역 (학습 내용 꼭지로 박지 말 것)

- **수강 흐름 안내 꼭지** — M1~M4 연결 설명은 메타 영역, 실제 학습 내용 X
- **세부 사용법** — 기능 X 사용법은 M3 영역
- **task 중심 설명** — 특정 task 연결은 M2 영역

### 작성 원칙 (★ topology 동일 반복 금지)

- 꼭지 수·제목·서술 워딩 모두 자율
- 같은 회사·다른 회사·다른 툴·다른 시점 case에서 **꼭지 제목·서술 워딩이 토씨 동일하게 박히지 않도록** 도구 특성·input.json 맥락에 맞는 자연 워딩으로 변형 작성
- 본 spec [필수 포함 학습 영역]의 항목명을 **그대로 꼭지 제목·본문으로 베끼지 말 것** (전형적 예시 베끼기 anti-pattern)
- 각 꼭지 본문 분량은 2~4줄

## 사용된 feature 상세 (꼭지 3의 근거 자료)

M1 꼭지 3에서 소개할 기능들을 아래와 같이 파악:

| feature | 요약 설명 | caveats |
|---|---|---|
| {feature_name_1} | {useful_for 기반 한 줄 요약} | {caveats or "-"} |
| ... | ... | ... |

※ M1에서는 feature의 "무엇을 하는가" 수준만 다룸. 세부 사용법·실습은 M3·M4 영역.

```

### 템플릿 채움 규칙

- **버전·조사일**은 tool-features.json에서 그대로 복사
- **주제 맥락 꼭지 (꼭지 2)**: 복수 주제(`topic_input.is_composite = true`)면 topic_elements 각각과의 관련성을, 단일이면 그 주제 하나와의 관련성 서술
- **feature_caveats 표시**: caveats가 null이 아닌 feature만 "※ 참고" 형식으로 노출

## 실행 절차 (5 Step)

### Step 1. 파일 로드
`{run_folder}/`에서 tool-features.json + input.json 로드. `top-tasks.json`이 있으면 추가 로드 (`topic_input`, `matched_topic`).

**툴별 순회** — `input.tools` 배열의 각 툴에 대해 Step 2~4 반복.

### Step 2. 주제 맥락 파악

top-tasks.json이 있으면:
- `topic_input.is_composite`, `topic_elements` 그대로 사용

없으면 (M1을 M4 체인보다 먼저 실행):
- `input.json.topic`을 자연어 그대로 사용
- 복수 주제 여부는 이 Skill 안에서 간단히 LLM으로 감지 (프롬프트 내부 step)

### Step 3. M1 모듈 구성 기획 생성 (LLM)

**생성 프롬프트**:

```
당신은 {tool_name}의 **M1(툴 소개) 모듈 구성 기획**을 작성합니다.
목적: 수강생이 이 툴을 처음 만나는 도입 단계에서 "이 툴이 뭐고 어떤 걸 할 수 있는지" 전체 그림을 잡도록.

**중요**: 구체 발화·강의 스크립트·슬라이드 대본 박지 말 것 (학습 주제·메인 학습 내용은 OK). **LD가 "이 모듈에 이런 꼭지가 들어간다"를 판단할 수 있는 모듈 구성 기획 수준**으로.

[툴 정보 — tool-features.json 기반]
- version_info
- features[] 전체 (name, description, useful_for, caveats)
- completeness_note

[주제 맥락]
- input.topic: {input.topic}
- is_composite: {true/false}
- topic_elements: {배열}

[회사·직무]
{company}, {role}

[M1 설계 원칙 — Phase 1 순수]
1. 배경 정보(버전·조사일·기능 수) 섹션은 간결하게
2. **강의 구성 제안 섹션이 주 본문** — 모듈 구성 기획 수준 (꼭지 수·제목·본문 워딩 자율)
3. 각 꼭지: 제목 + 다룰 내용·핵심 메시지 2~4줄
4. **구체 발화·강의 스크립트·슬라이드 대본 박지 말 것** — 수강생 발화, 슬라이드 대본, 구체 정의·구체 사용 원리·구체 설명문 X. 모듈 구성 기획 수준 (학습 주제·메인 학습 내용은 OK)
5. **bullet 작성 패턴**: 학습 동사 (이해·정리·판단·구분·도출·비교 등) + 영역 명세 + 콜론·괄호 뒤 1-2개 카테고리 자연 녹이기
6. **자연 녹이기 영역**: 도구 자체 속성 (도구 유형 일반 카테고리·도구 기능·인터랙션 패턴) + input.json source (company·role·topic)
7. **박지 말 것**: input.json.tools 내 다른 도구 비교 (각 도구별 별도 산출이라 모듈 영역 침범) / 구체 타사 제품명 거론 / 회사 사업 맥락 (Phase 3) / 임의 통계·인용·도입 사례 / 발화 스크립트
8. **세부 사용법 설명 금지** — "기능 X는 이렇게 쓴다"는 M3 영역
9. **수강 흐름 안내 꼭지 금지** — M1~M4 연결 설명은 실제 커리큘럼 bullet에 들어가지 않음
10. feature_caveats → "※ 참고: ..." 표시만 (대응 설계 X)
11. **구체 시간·분량 수치 금지** — "N분 강의" 등. Phase 2 영역
12. 복수 주제면 주요 기능 영역에서 주제별 주목 기능 간단 언급 가능
13. 사용 feature 상세 표는 LD 참고용, 간결하게
14. ★ **topology 동일 반복 금지**: 같은 회사·다른 회사·다른 툴·다른 시점 case에서 꼭지 제목·서술 워딩이 토씨 동일하게 박히지 않도록 도구 특성·input.json 맥락에 맞는 자연 워딩으로 변형. 본 spec [필수 포함 학습 영역] 항목명을 그대로 꼭지 제목·본문으로 베끼는 패턴 금지

[필수 포함 학습 영역 — 어떤 꼭지 형태·꼭지 수로든 반드시 다룰 것]
- 도구 정체 (이 AI가 어떤 종류의 도구인지 인식·정리하는 학습)
- 기본 사용 원리 (이 도구의 사용 패턴·인터랙션 인식·이해하는 학습)
- 주요 기능 개괄 (핵심 기능 카테고리 인식 — 세부 사용법은 M3 영역)

[출력 형식]
SKILL.md §"파일 구조" 템플릿 그대로. 학습 목표 / 툴 기본 정보 / 강의 구성 제안 / 사용된 feature 상세 순서.
```

### Step 4. 파일 저장
`m1-{tool_name_safe}.md`로 저장.

## 실패 처리

### 자동 재시도
LLM 호출 실패 시 최대 2회 재시도.

### Case A (LD 조치 가능)
- `tool-features.json`의 해당 툴 블록이 features 배열이 비어있음 → Skill 3 재실행 유도:
  ```
  ⚠ {tool_name}의 기능 정보가 부족합니다.
  M1 모듈 설계를 위해 Skill 3 (툴 기능 조사)을 먼저 재실행해주세요.
  ```

### Case B (시스템 오류)
- `tool-features.json` 파일 부재 → Skill 3 선행 요구 메시지
- `input.json` 부재 → Skill 1 선행 요구 메시지

## LD 피드백 처리

**기본 = 자동 진행** — Skill 8은 LD 출력 후 응답을 끊지 않고 Phase 1 조합으로 바로 이어간다 (STEP 5~8은 한 턴 안에서 흐르는 구간이라 LD 응답을 받는 지점이 아님 — Phase 1 조합 끝이 다음 주요 확인 포인트). 만일 LD가 능동적으로 개입하면 아래대로 반영:

- **"OK" / "진행해" / "계속"** → Phase 1 조합 단계로 진행
- **"꼭지 N을 다르게"** → 해당 꼭지만 재생성
- **"꼭지 추가/삭제"** → 재실행 (LD 요청 반영)
- **"feature 소개를 더 풍부하게 / 더 간결하게"** → 꼭지 3 재생성

Skill 8은 임의 변경 금지. LD 요청 시만 수정.

## 금지 사항

- **구체 발화·강의 스크립트·슬라이드 대본 박지 말 것** — "여러분은 이제..." 같은 강의 스크립트, 구체 정의·구체 사용 원리·구체 설명문 X. **모듈 구성 기획 수준**에 머물 것. Skill 5·6·7과 동일 원칙 (학습 주제·메인 학습 내용은 OK)
- **★ topology 동일 반복 금지** — 같은 회사·다른 회사·다른 툴·다른 시점 case에서 꼭지 제목·서술 워딩이 토씨 동일하게 박히지 않도록 변형. 본 spec [필수 포함 학습 영역] 항목명을 그대로 꼭지 제목·본문으로 베끼는 패턴 금지. 도구 특성·input.json 맥락에 맞는 자연 워딩으로 작성
- **★ input source 밖 임의 영역 박기 금지** — input.json.tools 내 다른 도구 비교 (각 도구별 별도 산출이라 모듈 영역 침범) / 구체 타사 제품명 거론 (input.json에 없는 도구 — hallucination) / 회사 사업 맥락 (Phase 3 영역) / 임의 통계·인용·도입 사례
- **세부 사용법·실습 방법 설명 금지** — "이 기능은 이렇게 쓴다"는 M3 영역. M1은 "무엇이 있는가·어떻게 쓰는 종류의 도구인가" 수준까지만
- **수강 흐름 안내 꼭지 금지** — M1~M4 연결 설명은 실제 커리큘럼 bullet로 옮기지 않는 메타 정보
- **단순 기능 나열 금지** — M1은 이론·기본 감 모듈. 기능 전체를 나열·설명하는 건 M3 영역
- **task 중심 설명 금지** — M1은 task-agnostic. 특정 task 연결은 M2 영역
- **구체 시간·분량 수치 금지** — "15분 소개", "2분 시연" 등. Phase 2 영역
- **feature_caveats 대응 설계 금지** — "※ 참고" 표시만 (대응은 Phase 3 영역)
- **missed_feature_estimates 참조 금지** — Skill 3·4가 이미 격리. M1에도 포함 X
- **★ 시수·수강생 수준·보안 제약 영역 산출물 박지 X 금지** — Phase 3 영역 (input.json.security/hours/level 영역 산출물 본문에 박지 말 것). LD가 input.json 자체 영역에서 별도 검토
- **tool-features.json에 없는 기능 창작 금지** — Skill 3 조사 결과만 사용
- **보안·시수·수준 고려 금지** (Phase 1 원칙)
- **top1 task 중심 설명 금지** — M1은 task-agnostic. 특정 task 강조는 M2 영역
